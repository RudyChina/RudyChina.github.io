Mysql索引底层数据结构与规范

周鸿宇诶 2020-09-17 MySql

# 导论：

  • 索引是帮助MySql高效获取数据的经过排序的数据结构

  • 索引存储在文件里,本质就是一个文件

  • 索引结构

    • 二叉树
    • 红黑树
    • Hash
    • BTREE

  • 磁盘存储原理

    • 寻道时间（速度慢，费时）
    • 旋转时间，用于定位数据扇区（速度较快）

# 红黑树

  • 对以上数据做同样操作，模拟图如下

  • 通过红黑树的自选平衡后，IO次数=树的深度+1，但是树的深度较二叉树要低。

  • 对于二叉树和红黑树做索引都有一个致命的缺点：当表的数据达到百万千万级别时，树的高度会无止境加深！！当我们要检索叶子节点时效率是十分低下的。

# HASH

  • HASH索引即把索引的值做hash运算后做文件的地址，很明显这种做法对于等值查找比较快（只需要对值进行一次hash运算就可以定位到文件地址）。但是对于范围查找效率比较低（数据在磁盘中比较分散）。

因此有了一个折中的数据结构也就是常用的B-TREE

# B-TREE

  • 度（Degree)：每个节点可以存储的最大数据个数。我理解的是为了控制二叉树，红黑树无法控制数的纵向高度。因此从横向的角度，让每个节点存储更多的数据，从而降低树的高度。

  • 每个叶节点具有相同的深度

  • 叶节点的指针为空

  • 叶节点的数据key从左至右依次递增

    那如果把度设成100，1000不应该更能减少树的高度吗？ 实际上这是没有意义的操作，我们的程序和数据的交互首先去内存查找，如果内存找不到再去磁盘找。而内存和磁盘的交互是有限制的，一般都是页的整数倍，即便我们将说有数据放到一个节点，但是我们的内存和磁盘的IO交互次数并不会减少。

因此Mysql使用了B-TREE的变种数据结构:B+TREE

# B+TREE

  • 非叶子节点不存储数据，只存储key（索引值）。节点的占用空间小了，那么度可以变大。总之思想就是尽可能的在一页的大小中存储更多的数据。从而降低树的高度又减少IO的次数。
  • 叶子节点不存储指针
  • 顺序访问指针，提高区间检索的性能

# InnoDB(聚集)

InnoDB的主键索引

InnoDB的主键索引叶子节点存储的是整行的数据

InnoDB的非主键索引

InnoDB的非主键索引叶节点存储的是索引行的主键

设计思想：

​ 避免重复维护数据，为了数据一致性而带来不必要的麻烦：如果主键索引和普通的索引都维护整行数据，这样需要考虑数据一致性的问题，同时浪费存储空间。

​ InnoDB的表数据文件本身就是按照B+Tree组织的一个索引文件，这也是为什么InnoDB一定要有主键。因此一定需要主键去维护。如果没有主键会自动选择一个不重复的列作为主键，实在没有这样的列也会自动生成一个主键列。

​ 同时InnoDB也推荐使用自增列作为主键:如果使用UUID作为主键，作为查询操作来说，我们需要对字符串转ASCII码做比较，而int类型不需要这个步骤。作为插入来说，UUID的插入不是连续的，很可能导致索引重排（叶子节点的分裂，移动等等）效率比较低。

# 联合索引的数据结构

​ 比较大小的顺序结合索引最左前缀，先比较第一个，如果能比较出大小直接跳过后续比较寻找下一个节点。如果相同则比较第二个索引，依次类推。

# EXPLAIN 执行计划

  • 在开发阶段对SQL的调优一般我采用的是Explain和慢日志的方式去查看Mysql是如何处理我的sql语句，再针对性的做查询优化。那对Explain查询出的列做一个简单的归纳。

# ID

  • 代表SQL的优先级，优先级越高，ID值越大。值相同从上往下优先级依次递增。ID为NULL的sql最后执行。

# select_type

  • SUBQUERY很显然代表的是一个子查询，-- PRIMARY代表的是复杂查询(最外层的复杂查询)。
  • 复杂查询：简单子查询，DERIVED派生表（from关键字后的子查询）,union查询。

  • union:代表的是union后的一个查询语句

  • union result：代表将union前后查询的结果集进行组装，等价于从union临时表检索结果的select

# type

  • 这一列表示访问类型，效率从高到低可表示为： System>const>eq_ref>ref>range>index>ALL

  • NULL：mysql能在优化阶段分解语句，执行的时候不需要再去访问表或者索引，如：在索引列中选取最小值，这种时候可以查找索引，不需要在执行时访问表。

  • SYSTEM,CONST：mysql能把查询的某部分优化成一个常量，用主键索引和唯一索引的所有列和常量进行比较时，表最多只能匹配一条数据，只需要读取一次。而CONST表示表里只有一条数据，是SYSTEM的特例

  • eq_ref：主键索引或者唯一索引联合查询的时候，最多只有一行匹配数据。

  • ref：不使用唯一索引，走普通索引的查询，可能会匹配多条数据行

  • range：范围查询出现在in()、between、>、<、>=等操作中。使用一个索引来检索给定范围的行。扫描的行数会比较多

  • index：会扫描全表索引，从索引中去查找需要的数据。从磁盘中读取。

  • ALL：会进行全表扫描，这种效率是最低的。从硬盘中读取。

# Extra

这一列展示的是额外信息

  • Using index：覆盖索引，查询的字段包含在了索引里面。并且Where筛选的条件是索引的前导列（联合索引的最左列），是sql性能高的表现。

  • Using Where：查询的列未被索引覆盖，where查询条件非索引前导列。需要去磁盘做IO操作

  • Using where Using index：查询列被索引覆盖，并且where筛选条件是索引列之一，但不是前导列，无法通过索引直接查询出需要的数据

  • NULL：查询条件未被索引全部覆盖，并且where筛选条件是索引的前导列，需要进行回表查询。

  • Using index condition：与Using Where类似，查询列未被索引覆盖，但是where的筛选条件是索引前导列的范围查询

  • Using Temporary：mysql需要创建一张临时表来处理查询，这种情况一般是要进行优化的。

    • mysql中distinct,group by,union,order by这种查询都会生成临时表，因此我们一般都需要给对应的查询字段加上索引，这样可以避免产生临时表。

  • Using filesort：mysql会对结果使用一个外部索引排序，actor的name字段没有索引，因此查询出来后会对结果集有一次文件排序的过程。但是film表的name字段建了索引，而索引是一个排好序的数据结构，因此不会再有文件排序的问题。

# sql规范

  • 全值匹配：用到联合索引，尽量把联合索引匹配条件用全。

  • 最佳左前缀法则：联合索引树的排序是根据索引的顺序进行比较的，因此一定要按照顺序使用索引。中间也不要断。

  • 不在索引列上做任何操作（计算、函数、（自动or手动）类型转换），会导致索引失效，全表扫描。

    • 联合索引的排序是根据索引的全量值排序的，我们对他进行截取等操作后跟全量值的排序是没什么关系的。

  • 存储引擎不能使用索引中范围条件右边的列

    • 同样结合联合索引底层数据结构，我们对第二列做范围查询后，那么查询就需要将满足第二列范围查询的数据全部捞出来再匹配第三个索引，但是第三个索引就没有走BTREE的结构，因此导致范围查询列后的索引列不会走索引查询。

  • 尽量使用覆盖索引，减少Select * 的操作

    • select * 会导致索引覆盖不到，需要进行回表IO操作。

  • 使用 != 或 <>的时候会导致索引失效

    • Btree的比较肯定是需要指定精确条件的，而通过不等于这种操作，没办法走Btree的查询。

  • IS NULL和IS NOT NULL没法走索引

  • LIKE的前置匹配会导致索引失效（like '%abc'）

    • Btree的排序是从前到后排序的，如果前置匹配（一个字符，多个字符），是没法走BTREE结构的。
    • 解决：使用覆盖索引，让他从索引字段去查
    • 注意：当覆盖索引指向字段是varchar(380)及以上时，覆盖索引会失效。
